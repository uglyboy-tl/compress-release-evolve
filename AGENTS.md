# 项目 AGENTS.md

## 必读

本项目基于「压缩、释放与进化」系统理论。**进行任何诊断或分析之前，必须先阅读白皮书（`academic-paper/white_paper.md`），理解完整的理论框架。** `theory/系统理论.md`、`theory/系统理论完整理解.md` 和 `theory/extensions/系统理论对学科范式的挑战.md` 可作为快速参考，但不能替代白皮书。

本项目的核心任务不是"分析热点事件"，而是**将系统理论运用在具体情境中**——产出文章和社交卡片。热点事件只是运用理论的场景之一。

---

## 核心理念

分析分为两个阶段：**诊断用熵，治疗用编码。**

诊断用熵的概念定位问题：什么东西被压缩了，熵转移到了哪里，哪些维度正在接近极限，为什么系统自己解决不了。治疗切换到编码进化的视角：系统需要什么样的新编码才能重新变得复杂而有韧性，而不是在旧框架内打补丁。

---

## 核心概念速查

以下概念是分析的基石。但记住：**概念是为叙事服务的，不是反过来。** 文章里不需要出现这些术语——读者只需要看到事实的冲突面。

| 概念 | 一句话 | 在文章里的形态 |
|------|--------|----------------|
| **熵** | 随机性。系统可能处于的状态越多，熵越大。 | “代价是什么呢？”——被挤压的东西去了哪里 |
| **压缩映射** | 控制 = 将复杂现象压缩为单一维度指标。总熵不会减少，只会转移。 | “Excel 一眼就能看到正收益” |
| **编码反转** | 手段变成目的，编码取代现实。 | 利润、增长、效率——这些是仪表盘上的数字，不是发动机本身 |
| **编码进化循环** | 熵减（形成新编码）→ 熵增（新旧组合）→ 新熵减 → 循环往复。 | 碰撞 → 新规则 → 新随机性 → 更新的规则 |
| **编码锁定** | 子系统围绕核心编码生长出高度耦合的依赖关系，难以自我革命。 | 三个月的牢笼——决策周期被资本市场锁死 |
| **散热带宽** | 系统释放熵的能力上限。产生速度 > 释放速度时必然崩溃。 | 会议室改工位，培训预算消失——这些维度正在失去散热能力 |

诊断用「熵」，治疗用「编码」。找到熵转移路径是诊断，找到编码进化循环是出路。

## 分析流程

### 收集素材

用 anysearch 搜索相关新闻、采访、评论，获取一手信息。让事件中的具体细节浮现出来——关键人物、关键场景、那些让人直觉"不对劲"的瞬间。素材越丰富，文章越有血肉。**不要急着套概念。**

### 诊断（用熵）

系统理论诊断的核心不是"套公式"，是持续追问三件事：

1. **什么被压缩了？** 哪里有单一维度被过度强调，把许多可能的做法压进了一条窄路？
2. **被挤走的复杂性去了哪里？** 熵不会消失。从被压缩的维度溢出来的随机性、成本、风险，落到了哪些不被度量的维度上？
3. **系统的进化在哪一环卡住了？** 编码进化循环（碰撞→新规则→新随机性→更新的规则）断在了哪里？

带着这三根尺子进入事件，你会发现它在你面前自然展开。**不用预设答案。** 有些事件的核心是"不该控的维度被控死了"（如 35 岁文章），有些是"该控的维度没控住"（如学术造假），有些是在熵的排放通道上出了问题。顺着事件本身的逻辑走，而不是把事件往框架里塞。

以下是一些在当代热点事件中**常见的**分析线索。它们不是什么标准步骤——只是当你追问"被压缩了什么"之后，最容易撞上的几种典型情况。遇上了可以用，遇不上别硬套：

- **编码反转**：一个原本是衡量工具的东西，被当成了系统追求的目标本身。比如论文数量从衡量学术产出的指标变成了科研的唯一目的。
- **自指闭环**：系统内部的反馈机制被锁在现有编码框架内，所有"纠偏"行为都在加固原有编码，无人能跳出框架审视它本身。
- **维度压缩偏误**：可度量的维度被天然放大，不可度量但对系统生存至关重要的维度被系统性忽视。
- **熵排放通道堵塞**：系统的散热带宽被一步步收窄——该释放熵的维度被压缩到无法承接，熵在内部堆积。
- **外部编码替代内部编码**：用可采购的标准化方案替代了组织在长期碰撞中沉积的默契和能力——表面效率提升，实则破坏进化循环。
- **弹性项被做成刚性约束**：本应是系统健康运行输出的指标（增长速度、利润率），变成了系统必须遵守的输入约束。
- **控制耦合**：上层路由和下层内部控制作用在同一个维度上，互相拉扯。比如资本市场季度考核（路由）和管理层经营决策（内部控制）叠加，既管不好也做不长。
- **路由失效 / 绕行**：母系统（路由功能）失效——该路由的不路由，不该路由的硬要路由。子系统自己找新的熵流出路：部门绕过瘫痪的IT系统自己搭工具，地方绕过失效的金融系统自己搞融资。

更重要的是，**你自己的视角可能会发现本文没列出的模式。** 系统理论提供的是几个底层追问，不是一种分类学。如果你在分析中看到一个不一样的、但切中事件要害的模式——那说明你找到了事件最准确的自然切口。

### 治疗（用编码）

诊断找到的熵转移路径，回答了"系统在哪里出问题"。治疗要回答的是"系统需要什么样的新编码才能重启进化循环"。**注意：这不是在问"怎么解决问题"，而是在问"系统需要什么样的编码进化循环"。** 解决方案必须是一个能自我维持的进化机制，不是一个一次性修补。

从这些方向切入：

- 旧编码是什么？它在哪个维度上锁死了系统的进化？
- 新的编码应**容纳**什么——不是"取消控制"，而是用更宽的框架替代更窄的框架？
- 新编码如何为熵的释放提供合法空间？哪些被挤压的维度需要重新获得呼吸权？
- 这个新编码能否启动一个可持续的进化循环——碰撞→新规则→新随机性→更新的规则？

**治疗阶段的关键限制：杠杆不能是"加一层新压缩映射"。** 不要提议"用 X 指标取代 Y 指标"来控制系统的运行——那是用同样的思路换一个刻度盘，本质是把控制加到了更多维度上。真正的杠杆是**调整奖惩结构**——让短视行为不再免费，让长期积累被看见，让熵泄漏的成本由释放者承担。这不是在加编码，是在改变编码之间生长的土壤。

### 组织叙事

诊断和治疗做完了，剩下的就是把它们写成文章。叙事结构的原则在下面「叙事结构」一节——那里是完整的指南。这里只强调一件诊断阶段和治疗阶段都容易忘的事：**叙事不是分析的翻译。** 不要把诊断结果编号列出来，然后每条配一个"这个现象在现实中表现为……"。整个诊断应该溶解在一个具体的场景里——一个人、一个瞬间、一次选择。35 岁文章开头的工程师点了通过、咽回了话、按下了取消键——这三个动作本身就是诊断的全部输出，不需要额外解释。

如果你发现自己写完后还在说"这就引出了系统理论中的压缩映射概念"——重写。读者不需要知道你在用哪个概念。他们只需要跟着那个工程师一起难受。

## 进化视角

系统理论的核心诊断维度是"进化卡在哪里"。这也是把系统理论分析和常规的"发现问题→解决问题"式分析区分开的关键——我们不只问"哪里坏了"，更问"系统的进化循环为什么没能转起来"。

### 编码进化循环

编码进化循环是诊断所有系统的通用框架：

1. **碰撞产生新编码**（熵减）— 系统在应对新情况时形成新的规则、默契、约束
2. **新编码与旧编码组合**（熵增）— 新规则与旧规则相互激发，产生新的可能性和新的随机性
3. **新一轮碰撞** — 新随机性暴露新问题，催生更新的编码

循环能转下去，系统就活着。循环停了，系统就死了——不管账面上多好看。

这个循环适用于任何正在演化的系统：一个团队、一家公司、一个行业、一个技术生态、一种社会制度。不同事件中"进化在哪一环卡住"的具体答案不同——有些是碰撞环节被压制，有些是组合环节被锁死，有些是新编码形成的通道被堵死。

**关键诊断问题：** 这个系统中，编码进化循环的哪一个半拍断掉了？是没人碰撞了，还是撞出来的东西没人接住，还是新规则固化后不再允许下一轮组合？

### 补充视角：嵌套系统

进化不是单层发生的。一个系统卡住，往往是因为上层系统的编码约束了下层系统的行为空间，而下层系统的反馈又无法传导回上层。

35 岁文章展示了一个典型的嵌套链条：资本市场（三个月季度考核）→ 企业管理层（降本增效决策）→ 员工个人（只做能写在简历上的事）。每一层的编码都在约束下一层的策略空间。

但嵌套不是唯一或必需的诊断路径。有些事件的进化瓶颈在系统内部（比如一个专业领域的方法论停滞），有些在跨系统接口（比如政策与市场的错配），有些在进化方式本身的僵化（比如一个学科长期没有方法论创新）。**让事件本身的逻辑决定你用哪把尺子。**

## 写作原则

### 叙事优先

先讲好故事，再用系统概念加深。不要写成分析报告。**35 岁文章的成功证明：** 读者不需要知道"熵""编码""嵌套系统"这些词——他们只需要看到那位活过裁员的工程师、他点通过的空说明栏、他咽回去的话、他桌面上一直开着的 LeetCode。这些场景本身就是理论的全部。用读者的语言说话——读者是谁，就用谁的语言。理论术语是为分析服务的，不是为展示服务的。

### 去掉 AI 味

删除填充短语（"值得注意的是""此外""然而"）、否定式排比（"不仅……而且……"）、破折号过度使用、粗体过度使用。读起来像人写的，不是像咨询报告。

### 中文引号

始终使用中文双引号（“”），不用「」。

### 让矛盾自己说话

读者不需要你告诉他“这很荒谬”、“这让人不舒服”。摆出事实的冲突面，读者自己会感受到。删掉所有替读者做情绪判断的形容词。35 岁文章里没有一处说“裁员很残酷”——它只描述了那位工程师按取消键的瞬间。

### 概念为叙事服务

不要先列概念再填例子。不要用"维度诊断""熵泄漏路径"这种咨询报告的语言。一个好的测试是：把文章中所有的系统理论术语删掉，文章还能不能站住？如果可以，概念放对了位置。

## 叙事结构

**章节排列遵循追问链。** 每一章回答上一章在读者心里种下的问题。排列不是任意的——找到读者读完每一章后最自然想问的下一个问题，那就是下一章。

**衔接靠逻辑，不靠桥句。** 不要写“让我们看看另一个角度”“但最容易搞反的还不是这个”这类外挂过渡。每章结尾的结论本身就应该让读者产生一个疑问，下一章开头直接回应那个疑问。35 岁文章的节奏是：留下的人怎么了 → 为什么会这样（发动机停了） → 停了之后公司做了什么（输血） → 为什么管理层这样做（三个月牢笼） → 这件事为什么重要（溢出，历史纵深） → 那怎么办。

**内容归类后合并。** 如果两个章节反复讨论同一个概念，合并重写，不要试图删重留异——本质上它们就是一个意思。

**大段重写是常态。** 当章节顺序调整或内容合并时，不要试图保留旧表述再拼贴——完全重新写一遍，往往更快，结果更连贯。

## 修订方法论

修改时要区分三种指令：

| 用户说 | 你应该做 |
|--------|----------|
| “这句需要改” | 看看是不是整段逻辑有问题，通常单句不顺是结构性问题的症状 |
| “这两章重复了” | 合并重写，不要试图删重留异——本质上它们就是一个意思 |
| “这段不太好” | 先问自己：这段要完成什么任务？如果现在的表达没完成任务，换一个表达方式，而不是修补现有措辞 |

**不要单句修改。** 用户指出一处表达问题时，检查周围段落是否存在更深层的结构性问题。频繁的单句修补会制造出衔接不上的碎片。

## 文章风格优化

分析文章写成后，按以下流程做风格优化。目标风格是兰小欢《置身事内》——**平实、理解而非指责、用叙事替代判断**。

### 风格特征一览

| 特征 | 说明 | 反面 |
|------|------|------|
| **理解结构，不指责个人** | 解释"为什么会这样"，不骂"谁做错了" | "从业者的傲慢""语重心长地告诉你" |
| **从具体案例切入** | 先讲一个故事、一个数据、一个人，再展开分析 | 直接抛出理论概念 |
| **口语化但不轻浮** | "冒出来了""水到渠成的事" | 论文腔（"从……的角度看……"）、滥用感叹号 |
| **短段落、短句** | 每段 2-4 句，手机屏幕一眼扫完 | 大段文字挤在一起 |
| **金句独立呈现** | 核心观点用引用块或加粗拎出来 | 埋在一大段话里被淹没 |
| **开放式结尾** | 收在问题或展望上，不强行给标准答案 | "综上所述"式的总结 |

### 优化流程

**第一步：通读，判断改动幅度。** 如果文章已有类比和叙事（如"发动机""编码"），微调即可，不要大拆大建。如果偏学术或偏指责，逐段重写语调。

**第二步：标题优化。** 公众号标题需要热词悬念 + 聊天式邀请；章节标题用问句代替陈述，用口语代替书面，去掉冒号结构。

**第三步：逐段语言优化。** 扫描全文，针对四个维度修改：
- **去掉道德判断和指责语气**："语重心长地告诉你"→"告诉你"，"傲慢"→"氛围"。原则：描述行为，不给人贴标签。
- **打散长段落**：超过 5 句的段落拆开。论点独立成段、金句独立成行、举例与结论分开。
- **调整加粗策略**：每节最多 1-2 处，全文 5-8 处。只加粗开篇悬念、核心论断、核心隐喻、过渡设问、结尾收束。
- **引入引用块**：全文 1-2 处，包装最生动的类比、最打动人的隐喻、意外感强的转折句。三枚举并列时用列表代替段落叙述。

**第四步：检查完整性。** 确认核心论点无遗漏、逻辑通畅、风格一致（不要前半段《置身事内》、后半段论文腔）。

### 实际案例对照

| 问题 | 优化前 | 优化后 |
|------|--------|--------|
| 全篇指责他人 | "从业者的傲慢""天真的傲慢" | "当信仰替代了问题" |
| 开头太急 | 直接切入理论 | 增加过渡："要回答这个问题，我们得先聊聊……" |
| 语气过重 | "典型的废话文学新版本" | 引用块"这就像说'跑得快的人跑得快'" |
| 标题太正式的冒号结构 | "从 LLM 到 Agent：一个自然的延伸" | "不只是说，还要去做" |
| 带评价的修饰 | "人总是有一种本能，可以将最赤裸的选择包装成不得已的苦衷" | "包装成不同的名字并不奇怪" |
| 感叹号式指责 | "你大概会愤慨：看！" | "读到这儿，你大概会觉得愤慨——" |

### 注意事项

1. **先读，再改。** 先理解作者的论证结构和核心观点，确保优化后不丢东西。
2. **尊重原文的"骨架"。** 不改结构、不改论点、不改论证顺序。优化只动语言、节奏、视觉呈现。
3. **公众号标题需要热词钩子。** 不要为了简约风格牺牲标题的信息量。折中方案：前半句设问钩子（保留热词），后半句聊天式邀请。
4. **保留原作者的独特表达。** 如果原文有好的类比、独特的叙事节奏，不要为了统一风格而抹掉。目标是**让好内容更容易被读懂**，不是把所有人的文章变成同一个模子。

## 工具使用技巧

### Playwright 渲染社交卡片

环境：`playwright-core` 已安装，系统上 `msedge` 可用（`/opt/microsoft/msedge/msedge`）。

```js
import { chromium } from 'playwright-core';

const browser = await chromium.launch({
  executablePath: '/opt/microsoft/msedge/msedge',
  headless: true,
  args: ['--no-sandbox', '--disable-gpu', '--font-render-hinting=none'],
});
```

**关键坑位：**

1. **`networkidle` 超时**：HTML 中引用了 Google Fonts 时，`networkidle` 永远不会触发（因为离线环境无法加载）。改用 `waitUntil: 'domcontentloaded'` 并手动等待：
   ```js
   await page.goto(`file://${HTML}`, { waitUntil: 'domcontentloaded', timeout: 60000 });
   await page.waitForTimeout(5000); // 等 WebGL canvas 和本机字体渲染
   ```

2. **viewport 设置**：需足够高以容纳所有竖向堆叠的 poster。如 10 张 1440px × 8 = 12000px，设置 `viewport: { width: 2400, height: 15000 }`。

3. **截图方式**：用 `element.screenshot()` 而非 `page.screenshot(clip)`，自动处理裁剪：
   ```js
   for (const [sel, name] of targets) {
     const el = page.locator(sel);
     await el.screenshot({ path: join(OUT, name), type: 'png' });
   }
   ```

4. **lightpanda 不可用**：本机 lightpanda 的 CDP 协议与 Playwright 不兼容，直接使用 msedge。

5. **本机字体**：优先使用本机 LXGW WenKai 替代 Google Fonts 的衬线字体，避免网络依赖：
   ```css
   --serif-zh: "LXGW WenKai GB Screen", "Noto Serif SC", serif;
   ```

### 社交卡片生成流程

1. 从 `guizang-social-card` skill 复制模板 `assets/template-editorial-card.html` 到 `social-cards/YYYY-MM-DD_event/index.html`
2. 选择风格：Editorial Magazine × E-ink（Indigo Porcelain 主题）或 Swiss International
3. 按 `references/layout-recipes.md` 选择每个页面的布局（M01-M16）
4. 每张卡一个 `<section class="poster xhs" id="xhs-XX">` — 所有卡写在同一个 HTML 里
5. 写入 render.mjs，运行 `node render.mjs`
